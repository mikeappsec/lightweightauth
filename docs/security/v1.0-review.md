# LightweightAuth v1.0 Security Review

**Status:** Pre-release self-review — M12 slice 9.
**Scope:** All code on branch `m12-v1-release` at HEAD.
**Reviewer:** Maintainer (self-review). Independent third-party review is
recommended before any deployment that handles unowned tenants' traffic.
**Threat model anchor:** [docs/DESIGN.md](../DESIGN.md) §"Trust Boundaries".

This review walks the v1.0 surface against the categories that historically
produce the most CVEs in identity/authorization gateways. Every finding is
either resolved on `m12-v1-release`, tracked as a known limitation in this
document, or filed as a non-v1.0 follow-up.

---

## 1. Cryptography

### 1.1 Primitives in use

| Use site                         | Primitive                          | Notes |
|----------------------------------|------------------------------------|-------|
| `pkg/identity/jwt`               | RS256, ES256, EdDSA, HS256         | HS256 disabled by default; opt-in via config. |
| `pkg/identity/dpop`              | ES256, EdDSA (RFC 9449)            | RS256 explicitly rejected per spec. |
| `pkg/identity/hmac`              | HMAC-SHA256                        | `hmac.Equal` constant-time compare. |
| `pkg/identity/mtls`              | TLS 1.2+, X.509                    | Cert validation delegated to `crypto/tls`. |
| `pkg/mutator/jwtissue`           | RS256, ES256, EdDSA                | Per-tenant key material; rotation supported. |
| `pkg/session`                    | AES-256-GCM (sealed cookies)       | Random 96-bit nonce per seal. |
| `internal/cache/decision`        | none (in-memory only)              | Keys are SHA-256 of canonicalized request. |

### 1.2 Findings & resolutions

- **F-CRYPTO-1:** All HMAC compares use `crypto/hmac.Equal` (constant-time).
  Verified by grep across `pkg/identity/{hmac,apikey}` — no `==` byte-slice
  compares on secret material. **Resolved.**
- **F-CRYPTO-2:** JWT `alg=none` is explicitly rejected before key lookup
  (see `pkg/identity/jwt/jwt.go` parser). **Resolved.**
- **F-CRYPTO-3:** Algorithm-confusion (HS256 signed with RSA pubkey) blocked
  by per-tenant `AllowedAlgs` allow-list; HS256 not in default allow-list.
  **Resolved.**
- **F-CRYPTO-4:** DPoP enforces `htm`+`htu` and `iat` window per RFC 9449.
  Replay window default 60s; nonce store optional (Redis/Valkey).
  **Resolved.**
- **F-CRYPTO-5:** AES-GCM nonces are generated via `crypto/rand` and never
  reused — sealing always allocates fresh nonce. **Resolved.**

### 1.3 Known limitations

- **K-CRYPTO-1:** Custom signing key formats are not supported; we accept
  PEM-encoded PKCS#8 / PKIX only. JWK Set ingestion is via standard library
  paths (`go-jose`) — no hand-rolled JWK parsing.
- **K-CRYPTO-2:** No FIPS 140-3 build mode in v1.0. Tracked for v1.1
  (use Go's `GOEXPERIMENT=boringcrypto` build).

## 2. Authentication correctness

### 2.1 Identifier modules

Each identifier (`apikey`, `hmac`, `mtls`, `jwt`, `dpop`, `oauth2`,
`introspection`) was fuzz-tested in M12 slice 6 (`pkg/identity/*/fuzz_test.go`,
~750k execs total). One real bug was found and fixed:

- **F-AUTHN-1:** `parseAuth` accepted `"a:"` (empty signature) as a valid
  compact-form HMAC credential. Fixed in slice 6; regression test in
  `pkg/identity/hmac/hmac_test.go::TestParseAuth_RejectsEmptySignature`.
  **Resolved.**

### 2.2 Conformance & contract testing

`pkg/module/conformance` (M12 slice 2) asserts every identifier:
- has a stable `Name()`,
- tolerates `nil` request,
- maps protocol-level errors to sentinel values (`ErrCredentialMissing`,
  `ErrCredentialInvalid`),
- does not retain references to caller-owned request memory,
- is goroutine-safe under 32-way concurrent invocation.

Built-in coverage validated: apikey, rbac, header-add (extending to all
identifiers is a v1.1 polish item).

### 2.3 Findings

- **F-AUTHN-2:** OAuth2 introspection responses are validated for `active=true`
  AND issuer match — a malicious introspection endpoint cannot impersonate
  another IdP because the configured issuer is enforced. **Resolved.**
- **F-AUTHN-3:** JWT `kid` lookup is bounded by per-tenant JWKS size limit
  (default 64 keys) to prevent unbounded-keyset DoS. **Resolved.**

### 2.4 Known limitations

- **K-AUTHN-1:** mTLS module trusts the network-level peer cert; if used
  behind an L7 proxy, operators MUST configure the XFCC parser instead.
  Documented in [docs/modules/mtls.md](modules/mtls.md).
- **K-AUTHN-2:** OAuth2 token introspection does not currently negative-cache
  *invalid* tokens — a flood of bogus tokens will hit the IdP. The
  `pkg/upstream` Guard bounds the blast radius (slice 8) but rate-limiting
  at the gateway is the recommended mitigation. Tracked for v1.1.

## 3. Authorization correctness

### 3.1 Policy engines

`pkg/authz/{rbac,opa,cel,openfga,composite}` are isolated behind the
`Authorizer` contract. Policy evaluation is **deny-by-default**: if no
matching rule exists, or if the engine errors, the decision is `DENY`.

### 3.2 Findings

- **F-AUTHZ-1:** Composite authorizer correctly short-circuits on first
  `DENY` (no further engines consulted), preventing a permissive engine
  from overriding a stricter one. Test:
  `pkg/authz/composite/composite_test.go::TestCompositeShortCircuitsOnDeny`.
  **Resolved.**
- **F-AUTHZ-2:** CEL programs are pre-compiled at config-load time; runtime
  evaluation cannot trigger fresh compilation. Compilation is bounded by
  CEL's own cost limits. **Resolved.**
- **F-AUTHZ-3:** OPA bundle ingestion validates SHA-256 over the bundle
  manifest before activation; tampered bundles are rejected. **Resolved.**
- **F-AUTHZ-4:** OpenFGA store / model IDs are validated against per-tenant
  allow-list to prevent cross-tenant tuple lookups. **Resolved.**

### 3.3 Decision cache

`internal/cache/decision` is keyed by the SHA-256 of the canonicalized
request (method + resource + sorted headers + tenant + identity). Two
findings:

- **F-AUTHZ-5:** Negative TTL (cache for `DENY`) defaults to 0 — denies are
  never cached unless operator opts in. Prevents a transient policy error
  from being amplified into a persistent block. **Resolved.**
- **F-AUTHZ-6:** Cache key includes `TenantId` — cross-tenant cache hits are
  structurally impossible. Verified by golden-config test in
  `tests/golden/`. **Resolved.**

## 4. Untrusted-input parsing bounds

All parsers exposed to network input have explicit size caps:

| Path                                     | Cap            | Source |
|------------------------------------------|----------------|--------|
| HTTP request body (Door A)               | 1 MiB default  | `internal/server/http.go` `MaxBodyBytes` |
| gRPC message (Door B)                    | 4 MiB default  | grpc-go default |
| JWT length                                | 8 KiB          | `pkg/identity/jwt` |
| API key                                   | 256 B          | `pkg/identity/apikey` |
| Header value (per-key)                    | 4 KiB          | `internal/server/http.go` |
| Header count (per-request)                | 100            | `internal/server/http.go` |
| OPA bundle                                | 16 MiB         | `pkg/authz/opa` |
| OpenFGA tuple page                        | 1000 entries   | `pkg/authz/openfga` |

Slice 6 fuzzers exercised the JWT, HMAC, DPoP, and XFCC parsers against
adversarial inputs without finding panics or unbounded allocations.

## 5. Resource exhaustion / DoS

### 5.1 Mitigations in place

- **Per-call upstream guards** (`pkg/upstream`): every external dependency
  (IdP, OPA, OpenFGA, Valkey) is wrapped in a circuit breaker + retry budget.
  Validated by chaos suite (slice 8): under a 100% failure storm with
  64 concurrent workers, only 5 of 2.15M calls reached the broken upstream;
  every other caller received a deterministic `ErrCircuitOpen`.
- **Decision cache** absorbs repeat traffic; default 60s positive TTL, 0s
  negative TTL.
- **Goroutine bounds:** every long-lived goroutine is owned by an explicit
  context-scoped lifetime (validated by goleak in slices 3, 5, 7).
- **Config reload:** xDS push uses a single-writer `Broker` (slice 3); the
  multi-writer extension is a documented post-v1.0 follow-up
  ([docs/DESIGN.md](../DESIGN.md) §M12 follow-ups).

### 5.2 Findings

- **F-DOS-1:** OpenTelemetry-go ≤1.40 had a baggage-header allocation DoS
  (GHSA-r3pj-fc6c-r6j8 / dependabot alert #1, high). Bumped `go.opentelemetry.io/otel`
  and friends to v1.41.0 in this slice. **Resolved.**
- **F-DOS-2:** `govulncheck ./...` (Go 1.26.2 toolchain) reports
  **no called vulnerabilities** at HEAD. **Resolved.**

### 5.3 Known limitations

- **K-DOS-1:** The built-in per-tenant rate limiter (`pkg/ratelimit`,
  M11) is local to each lwauth replica — token counts are not shared
  across replicas. With N replicas a tenant's effective limit is
  `N × rps`. Deployments needing strict aggregate enforcement should
  pair it with an L7 proxy rate limiter (Envoy local rate limit,
  NGINX `limit_req`). Distributed rate limiting tracked for v1.1.

## 6. Plugin trust boundary

`pkg/plugin/grpc` runs operator-supplied authorizers / identifiers /
mutators as **separate processes** over a Unix domain socket. The trust
contract is documented in [docs/DESIGN.md](../DESIGN.md) §"Plugin model":

- Plugins run **outside** the lwauth process — a panicking or crashing
  plugin cannot bring down the gateway; the upstream Guard turns
  repeated failures into a circuit trip, served as `DENY`.
- The plugin descriptor (`api/proto/.../plugin/v1/plugin.proto`) is locked
  by `tests/golden/TestPluginDescriptorLock` (slice 1) — wire-incompatible
  changes break CI.
- Plugin processes inherit the gateway's UID; operators should run lwauth
  as a non-root user with seccomp/AppArmor confinement applied uniformly.
- Plugins cannot issue their own decisions in isolation — every reply
  passes through the conformance contract checks (`pkg/module/conformance`).

### 6.1 Findings

- **F-PLUGIN-1:** Plugin RPC payloads are size-capped at 4 MiB (grpc default)
  and per-call deadline-bounded. **Resolved.**
- **F-PLUGIN-2:** Plugin identity is asserted by socket-path ownership
  (only the operator can place a plugin into the configured plugin dir);
  there is no in-band signature on plugin replies. Acceptable for v1.0
  (the plugin process is part of the operator TCB) but documented.

## 7. Multi-tenant isolation

Tenancy is enforced at three layers:

1. **Config layer:** `AuthConfig` is namespaced; the controller
   (`internal/controller`) refuses cross-namespace `IdPRef` references
   unless the operator opts in. Validated by envtest
   (`tests/envtest/idpref_propagates_test.go`, slice 4).
2. **Decision cache:** key includes `TenantId` (F-AUTHZ-6).
3. **Upstream guards:** one Guard per (tenant, upstream) pair — a noisy
   tenant trips its own breakers without affecting another tenant's
   traffic.

### 7.1 Findings

- **F-TENANT-1:** Plugin processes are **not** per-tenant by default —
  one plugin instance serves all tenants. Operators wanting strong
  per-tenant plugin isolation must run separate plugin processes and
  reference them per-tenant in config. Documented in
  [docs/modules](modules/).

These are explicitly **not** v1.0 blockers but are tracked for follow-up.
Ordering matches the reprioritized tiers in
[../DESIGN.md §7 Post-v1.0 queue](../DESIGN.md): **risk first, features
last.** The original IDs survive so existing references resolve.

| ID            | Item                                                                                                                                                                                                | Target |

| SEC-PROXY-1   | `lightweightauth-proxy` adapter now hands the engine `r.URL.RequestURI()` (preserving `RawQuery` for HMAC), reads inbound bodies once into a `--max-auth-body`-bounded buffer with a `MultiReader` replay to the upstream, refuses to bind a truncated prefix when the body exceeds the cap (over-cap requests still forward, but with `module.Request.Body == nil` so HMAC's deny-on-mismatch fires instead of being bypassed by a shifted prefix), and routes every error / deny body through a `publicReason` mirror of `internal/server/public_reason.go`. ### Tier A — hardening (v1.1)

| ID                  | Item                                                                                                                                                              | Target |
|---------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------|--------|
| F-PLUGIN-2          | The `grpc-plugin` adapter now supports an optional application-layer HMAC-SHA256 signature over plugin responses, carried as gRPC trailing metadata (`lwauth-sig` / `lwauth-kid` / `lwauth-alg`). Three modes: `disabled` (v1.0 default), `verify` (accept signed or unsigned, reject *bad* signatures), `require` (every response must be signed). A flood of identical-token retries during an IdP blip now coalesces to one upstream call per token per window instead of fanning out. The Guard circuit-breaker still owns the per-(tenant,upstream) coarse policy; this adds per-credential coalescing on top. | K-DOS-1             | Distributed (cross-replica) rate-limit aggregation backed by Valkey, keyed on `(tenant, bucket)` with a sliding window. Per-replica buckets remain the default.    | v1.1   |
| M10-PLUGIN-LIFECYCLE| Opt-in supervisor for `grpc-plugin` children: process exec via `os/exec`, periodic `grpc.health.v1.Health.Check` over the data-plane credentials, terminate-and-restart on `failureThreshold` consecutive failures, SIGTERM → `gracefulTimeout` → SIGKILL on Unix (Kill on Windows), exponential backoff (`initial * 2^n` capped at `maxBackoff`) with uniform ±`jitter`, `maxRestarts: 0` = unlimited. Engine startup blocks up to `startTimeout` for the first successful probe and surfaces failures as `ErrConfig`. Lives in new package [pkg/plugin/supervisor](../../pkg/plugin/supervisor/) (cross-platform via `signal_unix.go` / `signal_windows.go`); wired through [pkg/plugin/grpc/lifecycle.go](../../pkg/plugin/grpc/lifecycle.go).
